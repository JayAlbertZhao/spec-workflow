# Codex Project Notes

## Project

This repository is `spec-workflow`: a composable workflow/prompt system originally built for Claude Code projects.

The canonical Claude-facing instructions are still in `CLAUDE.md`. Codex should treat this file as the bridge layer and read `CLAUDE.md` before making workflow-level changes.

## Language

- Code, prompt files, agent files, and commits: English.
- User-facing conversation and status notes: Chinese.

## Operating Model

- Specs live under `specs/` and are orthogonal workflow concerns.
- `specs/route.md` maps a project brief to the specs that should be loaded.
- `scripts/bootstrap.sh` creates a new project from this base, initializes its own git repo, appends a project layer to `CLAUDE.md`, and prepares `brief/` plus `.agents/`.
- `.agents/` is the cross-session/project coordination area.
- Lines beginning with `>>` are user-directed instructions and must never be generated by agents.
- This repo is a workflow framework first. The Codex plugin manifest and scenario-specific `skills/` wrappers are packaging layers; reusable protocol remains in `specs/`.
- Skill naming follows `docs/design/skill-taxonomy.md`: `core-*`, `domain-*`, and `tool-*`.

## Important Discipline

- Never push to remote without explicit user request.
- Do not fabricate paths, APIs, or spec behavior; read the relevant file first.
- Preserve the distinction between base conventions and project-specific overrides.
- Keep reusable workflow rules in `specs/<name>/`; keep one-off project rules in the bootstrapped project's `CLAUDE.md`.
- If changing a spec protocol, update both the spec `CLAUDE.md` and any prompt templates it references.
- If adding or wrapping a spec, follow `specs/spec-authoring/CLAUDE.md`.
